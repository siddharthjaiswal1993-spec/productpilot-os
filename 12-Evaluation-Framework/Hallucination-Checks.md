# Hallucination Detection and Prevention

## Why Hallucination Is a Critical Risk

Hallucination in a product intelligence tool is not an inconvenience — it is a trust-killing failure mode. If a PM stakes their credibility on a priority recommendation backed by "12 enterprise accounts," and it turns out that number was generated by the LLM rather than sourced from actual CRM data, the PM is exposed and the product is done.

ProductPilot OS treats hallucination as a first-class safety concern, not a quality metric to optimize incrementally.

---

## Hallucination Taxonomy

Not all hallucinations are equal. ProductPilot OS defines three categories:

### Category 1: Factual Numeric Hallucinations (Critical)
The agent generates a specific number (account count, ARR value, percentage change, user count) that does not appear in retrieved evidence or CRM data.

**Example:** Brief says "23 enterprise accounts affected" but only 8 account names appear in retrieved evidence.

**Severity:** Critical — numbers are the most credible-sounding and most consequential type of claim in a prioritization brief.

**Detection:** Numeric claim extraction + cross-reference against evidence set. Every number in the output is checked against the retrieved evidence list.

### Category 2: Customer Attribution Hallucinations (Critical)
The agent attributes a need, complaint, or request to a specific customer that is not mentioned in retrieved evidence in that context.

**Example:** PRD says "GlobalTech's CTO specifically requested real-time sync" but GlobalTech does not appear in the evidence set for this feature.

**Severity:** Critical — using a specific enterprise customer's name to justify a decision, when that customer never said this, is a material misrepresentation.

**Detection:** Customer entity extraction + cross-reference against evidence set customer entities.

### Category 3: Contextual Inference Hallucinations (Medium)
The agent generates plausible but unverifiable inferences that are presented as factual claims.

**Example:** Brief says "this feature will reduce churn by 15-20%" — this is an LLM-generated estimate based on patterns in training data, not a claim sourced from any retrieved evidence.

**Severity:** Medium — inferences that are clearly labeled as estimates are lower risk; inferences presented as facts are higher risk.

**Detection:** LLM-as-judge eval scoring faithfulness of claims against retrieved evidence.

## Prevention Architecture

### Prevention Layer 1: Citation-Bound Generation
The agent system prompt requires every factual claim to be bound to a specific evidence chunk ID. Claims that cannot be bound to a chunk are prohibited:

```
System prompt constraint:
"You MUST NOT generate any of the following without a direct citation to a retrieved evidence chunk:
- Specific numbers (account counts, ARR values, user counts, percentage changes)
- Customer or company names
- Dates of events
- Quotes attributed to specific individuals
If you cannot cite the claim, omit it."
```

### Prevention Layer 2: Post-Generation Hallucination Check
After generation, before delivery to PM:

```python
def hallucination_check(output, evidence_set):
    # Extract all numbers from output
    numbers = extract_numbers(output)
    # Extract all customer/company names from output  
    entities = extract_entities(output)
    
    for number in numbers:
        if not any(number in chunk.text for chunk in evidence_set):
            flag_as_uncited(output, number, "NUMERIC_HALLUCINATION")
    
    for entity in entities:
        if not any(entity in chunk.text or entity in chunk.metadata.account_name 
                   for chunk in evidence_set):
            flag_as_uncited(output, entity, "ENTITY_HALLUCINATION")
    
    return flagged_output
```

### Prevention Layer 3: LLM-as-Judge Faithfulness Scoring
For a 10% sample of all outputs, a separate LLM evaluator scores faithfulness:

```
Prompt to judge LLM:
"Given the following evidence chunks [evidence] and the generated output [output]:
For each factual claim in the output, determine if it is:
A. Directly supported by the evidence (score: 1)
B. A reasonable inference from the evidence (score: 0.5, flag as INFERENCE)
C. Not supported by the evidence (score: 0, flag as HALLUCINATION)

Report: faithfulness_score = sum(scores) / total_claims
Report: any claims flagged as HALLUCINATION"
```

## Handling Detected Hallucinations

**Critical hallucinations (numeric, entity):**
- Remove the claim from the output
- Insert: "[Claim removed — source validation failed. PM verify.]"
- Log the hallucination in the eval database
- If the same type of hallucination occurs in the same output type >5% of the time, alert engineering immediately

**Medium hallucinations (inference):**
- Keep the claim but label it explicitly: "[AI estimate — not directly sourced from evidence]"
- Lower the confidence score's citation coverage component
- Log for LLM-as-judge evaluation

## Hallucination Rate Metric

```
Hallucination Rate =
  (Outputs containing at least 1 detected critical hallucination)
  ──────────────────────────────────────────────────────────────
  (Total outputs delivered to PM inbox)
```

**Target:** <2%  
**Critical alert threshold:** >5% over any 24-hour window  
**Measurement:** Combination of automated claim checking + LLM-as-judge sample

**Note on measurement completeness:** The automated check catches numeric and entity hallucinations reliably. The LLM-as-judge catches contextual hallucinations on sampled traffic. The true hallucination rate may be higher than measured — this is why the target is conservative (2%) rather than aspirational.

---

## PM Hallucination Reporting

PMs can report a suspected hallucination from the citation panel:
- Click "Mark as incorrect citation" on any citation
- System logs: which claim, which citation was flagged, PM ID, timestamp
- Flagged claims are reviewed by the trust team within 48 hours
- If confirmed as hallucination: added to golden dataset as adversarial example; prompt adjusted; eval case added

Every PM hallucination report is a high-value signal. The system should make it frictionless to report — and should respond visibly (PM notified when their report was reviewed and what action was taken).
